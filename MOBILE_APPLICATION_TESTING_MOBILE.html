1.4 Business Architecture (Mobile App)

  • Product: OnTime Ethiopia ‚Äì a mobile OTT/streaming application that provides live TV, shorts, and related content.
  • Core actors:
    • Mobile users (Android) accessing video content through the app.
    • Backend services (Django REST API) delivering authentication, content metadata, and streaming URLs.
  • Core backend: A Django REST Framework application exposing APIs for authentication, channels, live TV, playlist management, and shorts ingestion.
  • Mobile client: A Flutter-based Android application that interacts with these APIs over HTTPS.
  • Deployment: The mobile client connects to production or staging environments hosted by AI TECHNOLOGIES PLC; all communication with backend services is over TLS.

1.5 Data Flow Diagram (Text Summary)

  1. The OnTime Ethiopia mobile app (Flutter client) communicates with the backend exclusively over HTTPS.
  2. The app authenticates users for restricted features such as login and account management, securely storing necessary tokens or session data.
  3. It submits REST API requests to the Django backend (for example, /api/channels/, /api/live/, /api/channels/shorts/ready/feed/, /api/auth/login/).
  4. The backend reads and writes data from the application database (e.g. PostgreSQL), including user accounts, live metadata, channels, playlists, and shorts processing jobs.
  5. For media playback, the backend returns HLS playlist URLs. The app then streams audio/video from CDN or streaming endpoints over HTTPS.
  6. If enabled, background analytics and error logs may transmit telemetry data to backend monitoring services.

1.7 System Architecture Diagram with Database Relation (Text Summary)

  • Mobile Client Layer

1.9 Threat Model Overview

We prepared a threat model for the OnTime Ethiopia mobile application focusing on the following elements:

  • Assets:
    • User accounts and authentication credentials.
    • Access tokens and session identifiers stored on the device.
    • Streaming URLs and related metadata.
    • App configurations and feature flags.
  • Threat examples:
    • Network traffic interception (Man-in-the-Middle) if HTTPS is misconfigured.
    • Theft or compromise of sensitive information or stored tokens on the device.
    • Unauthorized API usage or bypassing access controls through manipulated requests.
    • Reverse engineering of the APK to reveal implementation details or API endpoints.
  • Mitigation examples:
    • Enforcement of HTTPS for all communication between the app and backend.
    • Server-side authorization checks for all protected operations.
    • Restricting and securing local storage of sensitive data, including minimal-duration tokens.
    • Applying Flutter build obfuscation/minification to reduce reverse-engineering risks.

5. Additional Mandatory Requirements

5.1 System Architecture

The OnTime Ethiopia mobile solution follows a modular architecture that clearly separates concerns across multiple layers:

  • Presentation layer: Flutter-based Android client responsible for UI, navigation, and basic input validation.
  • Application layer: Django REST API and Celery workers handling business logic, authentication, authorization, and content orchestration.
  • Data layer: PostgreSQL (or equivalent RDBMS) for structured data and object storage/CDN for media assets (HLS playlists, thumbnails, logos).
  • Integration layer: Connectors to external services such as YouTube/playlist sources, CDNs, and optional identity providers (e.g. Google).
  • Security layer: HTTPS/TLS, authentication/authorization, input validation, logging, and access control implemented primarily on the backend.
  • Monitoring layer: Application and infrastructure logs, error tracking, and basic metrics to observe system health.

The architecture is designed to support scalable deployment on cloud or on-premises environments, including:

  • Horizontal scaling of application servers behind a load balancer.
  • Containerization support for the backend and workers (e.g. Docker/Kubernetes) as part of CI/CD pipelines.
  • High availability through redundant instances and database backup/restore procedures.
  • Network segmentation between public-facing endpoints, application tier, and data stores.
